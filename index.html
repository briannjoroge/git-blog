EVERYDAY GIT

In the world of software development, change is constant. Code evolves rapidly, features are added, bugs are fixed, and improvements are made. To keep track of these changes and to collaborate effectively, developers rely on tools called version control systems.

What is a Version Control System ?

A Version Control System is a tool that helps developers track and manage changes to their code over time. There are several Version Control Systems and they include:

  • Helix Core
  • Mercurial
  • Subversion
  • AWS CodeCommit
  • Git

The above list is not comprehesive there are more version control Systems out there. In this blog I will discuss more on Git.

What is git ?

Git is a distributed version control system used to track changes in source code during software development. It allows multiple developers to work on the same project at the same time without interfering with each other’s work.

Having known what is Git, I will take you through the process of installing and configuting Git in your system. The first step will be checking whether git is installed in your system.

Checking if Git is installed

To check if git is installed in your system open the command line and run the command below:

git --version

If You get a response similar to the one above the you already have git installed in you system

If git is not installed and you are in a windows machines you will get a messsage like:

'git' is not recognized as an internal or external command, operable program or batch file.

if you are in a linux machine you will get some similar to this

To install Git we first need to download it. Visit the webpage HERE to download Git and make sure you download one that is compatible with your operating system.

After installing git, run the command git --version in the command line to be sure git is set up successfully in your system. If everything works as expected then we can proceed and configure git.

Intializing a Git repository

A Git repository is the central location for storing and managing all the files and their version history for a project. It allows developers to track changes, collaborate, and manage different versions of their codebase efficiently.

There are two types of Git repositories:

  1. Local Repository: A local repository exists on your computer and allows you to track changes, commit updates, and manage your project without needing an internet connection.
  2. Remote Repository: A remote repository is stored on a server or cloud-based platform like GitHub and is designed for collaboration, enabling multiple developers to push their changes, pull updates from others, and keep the project synchronized.

Before initializing a Git repository we first need to check if the current working directory is a Git repository, to do so, run the following command:

git status

Based on the above results, we can conclude that we are not in a git repository. We can now proceed to intializing one by running the command below:

git init

The hightled messsage is what we are looking for, it basically tell us that our repository was initialized successfully.

Adding files to the staging area

Once you’ve created a Git repository, the next step is to add your files to the staging area in preparation for committing.

The staging area in Git is an intermediate area where changes can be formatted and reviewed before committing them to the repository.

To add files to the staging area run the following command:

git add .

Tracked files vs Untracked files

Tracked files are files that have been added to the repository and are being monitored by Git. This includes files that have been committed and those that have been added to the staging area but not yet committed.

Untracked files are files that have been created in the working directory but have not yet been added to the staging area.

you can run git status to check the status of the files

The files in green are tracked files and have aready been added to the staging area ready for committing while the files in red are untracked files and have not yet been added to the staging area

Commiting changes

After adding your changes in the staging area, the next step is to commit the changes. To make a commit we run the following:

git commit -m "YOUR_COMMIT_MESSAGE"

The commit message should anser what the commit is doing eg adding a specific file, modifying the index.html, fixing a bug etc

Viewing past commits

To view past commits we run the command:

git log

the components of a git commit include:

  • hash - a unique identifier for the commit
  • commit message - purpose for the commit
  • author - the person making the commit
  • timestamp - time and date when the commit is made

Sometimes in repository can have hundrends of commits, to summarize the ouput of git log we add the flag --oneline

Going back to a specific commit

To go back to a specific commit we run the command:

Branches

Branching means you diverge from the main line of development and continue to do work without messing with that main line

Branches are essential for managing different versions of your codebase, allowing developers to work on separate tasks without affecting the main project.

The best use case for branching is when you want to work on new features, bug fixes, or experiments simultaneously without affecting the main codebase.

Creating a new branch

To create a new branch run the following command:

git branch BRANCH_NAME

Viewing all branches

To view all branches in your repository, you need to run the command:

git branch

the asterick (*) basically shows the current active branch, in simpler terms, it shows the branch you are currently working in.

Switching active branches

Times come when you want to switch from a the main branch to a testing branch or another branch and work on a feature or bug without affecting you main codebase. To switch active branches we basically run:

git switch BRANCH_NAME

After switching the active branch you can run the git branch command to confirm the new active branch shown by the asterick (*)

In my case I switched from the master branch to the testing branch

NOTE: After working in one branch make sure you add the changes to the staging area and commit the changes before switching to another branch to avoid carrying your changes with you as you switch to the other branch.

Merging a branch

Merging a branch in Git involves combining changes from one branch into another.

For example in the testing branch, am testing a new feature and now the feature is fully developed and I want to bring it over the the master branch where my main codebase is.

In this scenario I need to merge the branches, to do so I should run the following command:

git merge BRANCH_NAME

before running the command, since am bring changes from the testing branch into the master branch, i should first switch into the master branch then run the command.

git merge testing

The flow of command should be, in this case.

  • git switch testing : to switch to the testing branch.
  • work on the new feature, bug fixes etc. in this case create the file new-feature.js
  • git add . : to add the new files and changes in to the staging area.
  • git commit -m "COMMIT_MESSAGE" : to commit the changes in the stageing area.
  • git switch master : switch back to the master branch.
  • git merge testing : To bring changes in the testing branch over to the master branch.

The ls command is just there to show us the availability of the new-feature.js file, which only comes available in the master branch after merging it with the testing branch.

Deleting a branch

After working a branch and merging it with the master branch, it becomes of no use to still have it in our repository unless you want to work with it later. to delete a branch we simply run the following command

git branch -d BRANCH_NAME

In this case suppose i want to delete the testing branch, i should run git branch -d testing
